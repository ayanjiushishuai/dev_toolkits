# doc_integrate: 文档整合与同步维护

★ Insight ─────────────────────────────────────
**核心原则：代码是唯一的真**。
文档状态标记可能过时（开发中断、遗忘更新），但代码不会说谎。
本 skill 的职责是让文档追上代码，而不是让代码迁就文档。
─────────────────────────────────────────────────

---

## 概述

本 skill 是 ez-dev 开发流程的收尾环节。ez-dev 创建的 `.dev_doc/<feature>/` 文档（design.md、plan.md、changelog.md）
在开发完成后需要整合到对应模块的 README 中。本 skill 专注于：

1. **整合**：将散落文档合并到模块
2. **同步**：确保文档描述与代码一致
3. **归档**：处理过时的设计文档

---

## 核心流程

```
┌─────────────────────────────────────────────────────────────────────┐
│  1. 扫描散落文档                                                     │
│     ├── 扫描 .dev_doc/ 根目录的日期前缀文档（YYYY-MM-DD-*.md）        │
│     ├── 识别 ez-dev 遗留文档（.dev_doc/<feature>/）                   │
│     └── 构建《文档处理表》                                            │
│                                                                     │
│  2. 代码验证（核心步骤）                                              │
│     ├── 检查文档提到的关键文件是否存在                                 │
│     ├── 对比接口签名、方法实现是否与文档描述一致                      │
│     ├── 用 git log 确认代码实际变动时间                               │
│     └── 决策：整合 / 归档 / 保留                                     │
│                                                                     │
│  3. 整合执行                                                         │
│     ├── 已完成且代码验证通过 → 合并到模块 README                     │
│     ├── 代码已实现但文档过时 → 更新文档后合并                         │
│     ├── 设计已废弃或重复 → 归档到 .archive/                          │
│     └── 更新模块 CHANGELOG.md                                        │
│                                                                     │
│  4. 同步维护                                                         │
│     ├── 修正文档中与代码不一致的描述                                 │
│     └── 记录同步修正到 CHANGELOG                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 决策树：如何处理每份文档

```
文档
  │
  ├─ 是否提到具体代码文件或方法？
  │   │
  │   ├─ 否 → 检查是否为核心文档（architecture.md 等）
  │   │        ├─ 是 → 保留在根目录
  │   │        └─ 否 → 归档
  │   │
  │   └─ 是 → 代码验证
  │        │
  │        ├─ 代码存在 → 提取文档关键内容，合并到模块 README
  │        │              （不是归档，是真正合并内容！）
  │        │
  │        ├─ 代码不存在
  │        │   ├─ 文档日期 < 30天 → 保留在根目录（可能需要开发）
  │        │   └─ 文档日期 >= 30天 → 归档
  │        │
  │        └─ 无法判断 → 用 git log 确认
  │
  └─ 特殊情况
       │
       ├─ .dev_doc/<feature>/ 目录 → ez-dev 遗留
       │    → 已完成：提取关键内容合并到模块 README
       │    → 未完成：保留在原目录
       │
       ├─ 空文档 → 立即归档
       │
       └─ 归档条件（严格限定）
            ├── 内容已被其他文档完全覆盖
            ├── 设计已废弃，不再需要
            └── 代码不存在且文档日期 >= 30天
```

★ Insight ─────────────────────────────────────
**归档是最后手段，不是默认处理**。
"代码已实现" + "文档有内容" = 必须整合（提取内容合并到 README）
只有"真正废弃或完全重复"的才归档。
─────────────────────────────────────────────────

## 状态判断规则

**文档状态标记只是参考，代码才是最终依据。**

| 文档标记 | 实际判断 |
|----------|----------|
| `状态：**已完成开发**` | ✅ 代码验证后整合 |
| `状态：**开发中**` | ⚠️ 检查代码是否完成，完成则整合 |
| `状态：**方案设计**` / `状态：**待开发**` | 代码验证：存在则更新后整合，不存在则归档 |
| 无状态标记 | 视为"未完成"，按代码验证处理 |
| 30天以上旧文档无完成标记 | 默认归档，除非代码验证通过 |

## 整合策略

### 小变动（追加到 README）

- bugfix
- 单个方法改动
- 配置变更

→ 在模块 README 对应章节下追加变更记录

### 大变动（新增章节）

- 新增 provider
- 架构重构
- 多文件联动

→ 新增独立章节或独立文档（README 引用）

### 设计文档（保留引用）

- 完整架构图、流程图
- 长期迭代维护的设计
- 跨模块复杂设计

→ 移动到模块目录，README 添加「相关设计」引用

## 代码验证方法

### 1. 文件存在性检查

```bash
ls <文档中提到的文件路径>
```

### 2. 方法签名对比

```bash
grep -n "def <方法名>" <文件路径>
```

### 3. Git 历史确认

```bash
git log --oneline --since="<文档日期>" -- <相关代码目录>
```

## ez-dev 遗留文档处理

ez-dev 为每个功能创建 `.dev_doc/<feature>/`，包含：

| 文件 | 处理方式 |
|------|----------|
| `status.md` | feature 已完成则归档，未完成则保留 |
| `design.md` | 整合到模块 README，保留引用 |
| `plan.md` | 归档 |
| `changelog.md` | 合并到模块 CHANGELOG |

## CHANGELOG 格式

```markdown
# 模块名 CHANGELOG

## [未整合] YYYY-MM-DD

### 文档变更
- 合并 `2026-03-30-xxx-design.md` 到 README
- 更新 `advanced_search()` 接口描述以匹配 v6.0 实现

### 代码变动（git log）
- `extra_function/utils/vector_stores/providers/local_vector_store.py`
  - 实现 `advanced_search()` 多向量字段支持

### 同步记录
- 文档与代码不一致处已修正：`LocalVectorStore._extract_vector` 逻辑
```

## 执行步骤

### Step 1: 扫描散落文档

排除：architecture.md、status.md、todo.md、work_done.md、.archive/、.update_doc_state/、开发文档编写原则/

### Step 2: 代码验证（每份文档必须执行）

1. 读取文档，提取关键文件路径
2. 检查文件是否存在
3. 对比接口签名
4. 用 git log 确认代码变动时间
5. 输出决策建议

### Step 3: 执行整合

**代码存在且验证通过 → 合并**
1. 移动文档到模块目录
2. 更新模块 README（追加章节或引用）
3. 更新模块 CHANGELOG

**代码不存在或已废弃 → 归档**
1. 移动到 `.archive/`
2. 记录到 CHANGELOG

### Step 4: 输出报告

```markdown
# 文档整合报告 - YYYY-MM-DD

## 整合统计
| 状态 | 数量 |
|------|------|
| ✅ 已整合 | X |
| 🗃️ 已归档 | X |

## 整合详情
[每份文档的处理结果和理由]
```

---

## 注意事项

1. **不要生成"待开发"列表** - 每个文档必须有明确处理结果
2. **代码是最终依据** - 文档与代码不符时，以代码为准更新文档
3. **不要保留在 limbo** - 不要说"待后续处理"，要立即决策
4. **git log 是好帮手** - 可以确认代码实际实现时间
5. **不删除任何文档** - 只移动或归档，保留历史痕迹
