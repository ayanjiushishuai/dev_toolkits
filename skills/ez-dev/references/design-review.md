# 方案自审清单 & 独立设计 Review 规范

> 本文档定义 Phase 1.5（方案自审）和 Phase 2.5（独立设计 Review）的 Review 机制和检查清单。
>
> **核心原则**：Review Agent 必须以"接手者"视角进行审查，不持有前序 Phase 的上下文，独立、严格、不带偏好。

---

## 问题分级标准

**致命问题（Critical）** — 必须修复，Review 不通过：
- 核心功能无法正常工作
- 安全漏洞或数据泄露风险
- 资源泄露（连接、文件、内存）
- 破坏向后兼容性
- 严重的性能问题

**非致命问题（Non-Critical）** — 记录即可，不阻断 Review：
- 代码风格不一致
- 边界条件遗漏（有合理兜底）
- 文档不完整（代码本身可理解）
- 重复代码（不导致 bug）
- 未来优化建议

---

## 独立 Review 通用机制

### 执行方式

**Review Agent 启动时：**
- 不自动加载前序 Phase 的对话上下文
- 只读取必要的文档：`requirement.md`（如有）、`.dev_doc/<feature>/design.md`、`.dev_doc/<feature>/plan.md`
- 以"刚接手这个需求的新开发者"视角进行审查

**Review 结论：**
| 结论 | 含义 | 后续动作 |
|------|------|----------|
| ✅ 通过 | 无严重问题或已知可接受的风险 | 进入下一 Phase |
| ❌ 不通过 | 发现需要修正的问题 | 返回对应 Phase 调整后，重新执行独立 Review |

**Review 失败处理流程：**
```
Review Agent 报告问题
    ↓
返回对应 Phase（Phase 1 → Phase 1，Phase 2 → Phase 2）
    ↓
AI 根据 Review 意见调整方案/设计
    ↓
重新执行独立 Review（必须再次通过）
```

## Phase 1.5: 方案自审清单

**触发时机：** brainstorming 确认需求后，生成初始方案，提交用户确认前

**核心心态：** "这个方案能安全交付吗？" — 而不是"我能找出多少问题"

**非致命问题处理原则：**
- 发现非致命问题 → 记录在案，继续通过
- 如果自纠后只剩非致命问题 → 直接通过，不重新 Review
- 只有致命问题才需要触发下一轮 Review

### 自审清单（精简版）

```markdown
# 方案自审报告

## 1. 致命问题检查
| 检查项 | 结果 | 说明 |
|--------|------|------|
| 核心功能是否可实现 | ✅/❌ | |
| 是否有安全/数据泄露风险 | ✅/❌ | |
| 是否破坏向后兼容性 | ✅/❌ | |
| 是否有资源泄露风险 | ✅/❌ | |

## 2. 模块影响评估
| 模块 | 改动点 | 影响等级 |
|------|--------|----------|
| [模块A] | [改动内容] | 高/中/低 |

## 3. 潜在风险识别
| 风险点 | 可能性 | 影响 | 应对 |
|--------|--------|------|------|
| [具体风险] | 高/中/低 | [影响] | [方案] |

## 4. 非致命问题记录（不阻断Review）
- [列出发现的非致命问题，但不影响通过决策]

## 5. 结论
- [ ] 无致命问题，或致命问题已有应对方案
- [ ] 可以提交用户确认
```

### 阶段声明

```
🔍 进入 Phase 1.5: 方案自审
正在评估方案能否安全交付...
```

## Phase 2.5: 独立设计 Review

**触发时机：** Phase 2 设计方案完成，用户批准后

**Review Agent 任务：** 以"接手者"视角审查设计方案是否满足需求、是否可实现、是否存在遗漏

**文档输入：**
- `.dev_doc/<feature>/requirement.md`（如存在）
- `.dev_doc/<feature>/design.md`
- `.dev_doc/<feature>/plan.md`

### 设计 Review 清单（精简版）

```markdown
# 设计 Review 报告

## 1. 需求覆盖检查
- [ ] 每个功能点都有对应实现方案？
- [ ] 没有需求遗漏？

## 2. 致命问题检查
| 检查项 | 结果 | 说明 |
|--------|------|------|
| 接口设计是否破坏兼容性 | ✅/❌ | |
| 是否有资源泄露风险 | ✅/❌ | |
| 是否有严重性能问题 | ✅/❌ | |

## 3. 模块边界
- [ ] 模块职责划分清晰？
- [ ] 无循环依赖？

## 4. 非致命问题记录（不阻断Review）
- [列出发现但不影响交付的问题]

## 5. 结论
- [ ] 无致命问题
- [ ] 设计可通过
```

### 阶段声明

```
🔍 进入 Phase 2.5: 独立设计 Review
正在以接手者视角审查设计方案...
```
