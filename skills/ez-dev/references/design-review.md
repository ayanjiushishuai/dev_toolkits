# 方案自审清单 & 独立设计 Review 规范

> 本文档定义 Phase 1.5（方案自审）和 Phase 2.5（独立设计 Review）的 Review 机制和检查清单。
>
> **核心原则**：Review Agent 必须以"接手者"视角进行审查，不持有前序 Phase 的上下文，独立、严格、不带偏好。

---

## 独立 Review 通用机制

### 执行方式

**Review Agent 启动时：**
- 不自动加载前序 Phase 的对话上下文
- 只读取必要的文档：`requirement.md`（如有）、`.dev_doc/<feature>/design.md`、`.dev_doc/<feature>/plan.md`
- 以"刚接手这个需求的新开发者"视角进行审查

**Review 结论：**
| 结论 | 含义 | 后续动作 |
|------|------|----------|
| ✅ 通过 | 无严重问题或已知可接受的风险 | 进入下一 Phase |
| ❌ 不通过 | 发现需要修正的问题 | 返回对应 Phase 调整后，重新执行独立 Review |

**Review 失败处理流程：**
```
Review Agent 报告问题
    ↓
返回对应 Phase（Phase 1 → Phase 1，Phase 2 → Phase 2）
    ↓
AI 根据 Review 意见调整方案/设计
    ↓
重新执行独立 Review（必须再次通过）
```

## Phase 1.5: 方案自审清单

**触发时机：** brainstorming 确认需求后，生成初始方案，提交用户确认前

**Review Agent 任务：** 自己挑刺，找出潜在问题，分析影响

### 自审清单

```markdown
# 方案自审报告

## 1. 涉及模块分析
| 模块 | 文件/类 | 改动点 | 影响等级 |
|------|---------|--------|----------|
| [模块A] | [文件路径] | [改动内容] | 高/中/低 |

## 2. 代码级影响评估
- [ ] 读取相关模块代码，分析改动点的调用链
- [ ] 识别会影响的上下游接口
- [ ] 检查是否有隐藏的依赖关系

## 3. 潜在 Bug 风险
| 风险点 | 可能性 | 影响 | 规避方案 |
|--------|--------|------|----------|
| [具体风险] | 高/中/低 | [影响描述] | [如何规避] |

## 4. 新增风险识别
- [ ] 是否有新的外部依赖引入
- [ ] 是否影响现有功能的兼容性
- [ ] 是否有性能/并发/安全风险

## 5. 方案问题挑刺
### 5.1 边界条件是否处理完整？
- [具体边界问题]

### 5.2 错误处理是否充分？
- [具体的错误场景遗漏]

### 5.3 设计是否存在冗余或过度设计？
- [具体问题]

### 5.4 测试覆盖是否存在盲区？
- [具体测试缺失点]

## 6. 优化建议
| 问题 | 优化方案 | 优先级 |
|------|----------|--------|
| [问题1] | [优化1] | 高/中/低 |

## 7. 最终方案确认
- [ ] 上述问题已全部处理或已评估可接受
- [ ] 方案可以提交用户确认
```

### 自审流程

```
1. 阅读相关模块代码（Glob/Grep 追踪调用链）
2. 填写自审清单
3. 如发现问题 → 返回方案设计进行调整
4. 自审通过 → 输出 design-review.md → 进入用户确认
```

## Phase 2.5: 独立设计 Review

**触发时机：** Phase 2 设计方案完成，用户批准后

**Review Agent 任务：** 以"接手者"视角审查设计方案是否满足需求、是否可实现、是否存在遗漏

**文档输入：**
- `.dev_doc/<feature>/requirement.md`（如存在）
- `.dev_doc/<feature>/design.md`
- `.dev_doc/<feature>/plan.md`

### 设计 Review 清单

```markdown
# 设计 Review 报告

## 1. 需求覆盖检查
- [ ] 设计方案是否覆盖了需求中的每个功能点？
- [ ] 是否有需求遗漏或理解偏差？
- [ ] 功能点的实现方案是否明确？

## 2. 接口设计检查
- [ ] 对外接口（API、类、方法）签名是否清晰？
- [ ] 接口是否与现有代码风格一致？
- [ ] 是否有可能破坏向后兼容性的变更？

## 3. 数据模型检查
- [ ] 数据结构变更是否影响现有存储？
- [ ] 是否有数据库迁移需求？
- [ ] 数据流是否清晰可追踪？

## 4. 模块边界检查
- [ ] 模块职责划分是否清晰？
- [ ] 模块间依赖关系是否合理？
- [ ] 是否有循环依赖风险？

## 5. 异常处理检查
- [ ] 关键操作是否有错误处理？
- [ ] 异常是否向上层暴露了足够的上下文？
- [ ] 是否有资源泄露风险（连接、文件等）？

## 6. 性能考虑
- [ ] 是否有明显的性能瓶颈？
- [ ] 大数据量场景是否考虑？
- [ ] 是否有不必要的重复计算或 IO？

## 7. 测试可行性
- [ ] 设计是否可测试？
- [ ] 是否有难以测试的死角？
- [ ] 是否需要新增测试基础设施？

## 8. 文档完整性
- [ ] 设计文档是否清晰可执行？
- [ ] 关键决策是否有记录？
- [ ] 是否有待确认的技术选型？

## 9. 风险评估
| 风险点 | 可能性 | 影响 | 应对 |
|--------|--------|------|------|
| [具体风险] | 高/中/低 | [影响] | [方案] |

## 10. Review 结论
- [ ] 通过 / 不通过
- [ ] 如不通过，列出必须修正的问题
```
