# BUG_FIX: Bug修复开发流程

★ Insight ─────────────────────────────────────
这个skill是严格的线性流程，每个Phase必须按顺序执行。
与ez-dev的核心区别：前期不需要高强度沟通，只需提供错误信息和相关模块。
重点在于：分析原因→定位代码→创建测试复现→确认后生成修复计划。
─────────────────────────────────────────────────

---

## 独立 Review 机制

### 通用规则（适用于所有 Review 节点）

**Review Agent 启动方式：**
- 启动独立 Subagent，不加载前序 Phase 的对话上下文
- 只读取必要的文档文件（`.dev_doc/<bug>/` 下的对应文档）
- 以"刚接手这个Bug的新开发者"视角进行审查

**Review 结论：**
| 结论 | 含义 | 后续动作 |
|------|------|----------|
| ✅ 通过 | 无严重问题 | 进入下一 Phase |
| ❌ 不通过 | 发现需要修正的问题 | 返回对应 Phase 调整后，**重新执行独立 Review** |

**失败循环：** Review 不通过 → 调整 → 重新 Review → 通过 → 继续。任何 Phase 的 Review 都必须通过才能进入下一 Phase。

### Review 节点一览

| Phase | Review 内容 | 输入文档 | 并发 |
|-------|-------------|----------|------|
| Phase 1.5 | 独立Bug分析 Review | bug-analysis.md、相关代码 | |
| Phase 2.5 | 独立修复计划 Review | fix-plan.md、bug-analysis.md | |
| Phase 3.1 + 3.2 | 计划完整性 + 合规门禁 | plan.md、fix-plan.md | ✅ 并发 |
| Phase 4.1 | 独立代码 Review | worktree 代码 | |
| Phase 5 | 验证审查 | worktree 代码、测试结果 | ✅ 独立 Subagent |
| Phase 5.1 | 用例生成 | plan.md、fix-plan.md、worktree 代码 | ✅ 独立 Subagent |
| Phase 5.5 | 用例质量审查 | plan.md、tests/ 用例 | ✅ 独立 Subagent |

---

## ⚠️ CRITICAL RULES

**以下规则无论任何情况都必须遵守：**

| 规则 | 说明 | 违规后果 |
|------|------|----------|
| **Worktree 强制** | 所有代码开发必须在 worktree 中进行，禁止直接在本地分支写代码 | 主分支污染 |
| **Review 强制** | 所有 Review 节点必须由独立 Subagent 执行，失败必须返回调整 | 质量无保证 |
| **Phase 顺序** | 禁止跳过任何 Phase，每个 Phase 都必须完整执行 | 流程残缺 |
| **Phase 声明** | 进入新 Phase 时必须向用户声明当前阶段 | 用户失去掌控 |
| **测试复现优先** | 在生成修复计划前，必须先创建测试用例复现Bug | 修复无验证 |
| **文档存储** | superpower skill 产出文档必须存储在 `.dev_doc/<bug>/` 下 | 文档散落 |
| **merge 用户手动** | merge 操作由用户执行，AI 不自动合并 worktree | 人工确认缺失 |

---

## 🔄 自纠闭环原则

**核心改变：Review 失败后立即自纠，不再等用户确认。**

| 需要用户确认的节点 | 不需要用户确认（自主闭环） |
|--------------------|---------------------------|
| Phase 1 Bug分析确认（Bug场景/原因/复现步骤） | Phase 1.5 Review 失败 → 自纠 → 重新 Review |
| Phase 2.5 修复计划通过 | Phase 3.1/3.2 Review 失败 → 自纠 → 重新 Review |
| **重大方向变动**（需用户决策） | Phase 4.1 Review 失败 → worktree 修复 → 重新 Review |
|  | Phase 5.5 Review 失败 → 自纠 → 重新测试 → 重新 Review |

**自纠循环规则：**
- Review 失败 → 立即根据 Review 意见自纠 → 重新执行 Review
- 自纠后仍然失败 → 继续自纠（最多 3 次循环）
- 3 次后仍失败 → 记录问题，暂停并通知用户（说明卡点，请求决策）

## Phase 1: Bug分析

**沟通程度:** 低（仅获取基本信息）

★ Insight ─────────────────────────────────────
Bug分析与需求分析不同：
- 不需要高强度沟通
- 只需要：报错信息 + 场景描述 + 相关模块
- 重点在于：分析原因、定位代码、创建测试复现
─────────────────────────────────────────────────

**任务:** 基于用户提供的信息，分析Bug原因，定位问题代码，创建测试用例复现Bug

**步骤:**
1. **收集信息**（用户已提供）
   - 报错信息：错误类型、错误消息、堆栈跟踪
   - 场景描述：发生Bug的操作步骤
   - 相关模块：用户指出的可能相关模块

2. **分析Bug原因**
   - 读取相关模块代码
   - 追踪调用链，找到问题根源
   - 识别具体的错误行/函数

3. **创建测试用例复现Bug**
   - 在 `tests/` 目录下创建测试用例
   - 测试用例必须能够**复现用户报告的Bug**
   - 运行测试，确认报错信息与用户报告一致

4. **确认复现**
   - 测试运行后产生与用户报告相同的错误
   - 记录复现步骤和预期行为 vs 实际行为

**阶段声明:**
```
🔍 进入 Phase 1: Bug分析
正在分析Bug原因、定位问题代码...
```

**文档存储:** `.dev_doc/<bug>/bug-analysis.md`

**完成标准:** 
- Bug原因已定位
- 测试用例已创建并成功复现Bug
- 用户确认Bug分析结果

**沟通程度:** 低（独立 Subagent 执行）

**触发时机:** Phase 1 Bug分析完成，用户确认后

**任务:** 启动独立 Subagent，审查Bug分析是否准确、测试复现是否有效

**执行方式:**
```

- 角色：刚接手Bug的新开发者
- 只读取：.dev_doc/<bug>/bug-analysis.md、相关模块代码、测试用例
- 不加载：Phase 1 的对话上下文
- 任务：审查Bug分析报告和测试复现有效性
```

**Review 检查清单:** 参见 `references/bug-review.md` 的 Phase 1.5 清单

```
🔍 进入 Phase 1.5: 独立Bug分析 Review
```

- Review Agent 报告问题 → 修正Bug分析和测试用例 → 重新执行 Phase 1.5 Review

**完成标准:** 独立 Subagent 给出 ✅通过 结论

## Phase 2: 修复计划

**沟通程度:** 中

★ Insight ─────────────────────────────────────
修复计划与设计方案不同：
- 重点是修复策略，不是新功能设计
- 必须包含：根因分析、修复方案、测试验证计划
─────────────────────────────────────────────────

**文档存储:** `.dev_doc/<bug>/fix-plan.md`

**阶段声明:**
```
📝 进入 Phase 2: 修复计划
正在制定Bug修复计划...
```

**修复计划要求:**
```
## Bug概述
- **报错信息:** [用户报告的错误]
- **影响范围:** [Bug影响了哪些功能]

## 根因分析
- **根本原因:** [导致Bug的代码问题]
- **触发条件:** [什么情况下会触发这个Bug]

## 修复方案
### 方案 A: [方案名称]
- **修复思路:** [如何修复]
- **涉及文件:** [需要修改的文件]
- **修改内容:** [具体修改点]

## 测试验证计划
- [ ] 原测试用例必须仍然失败（确认复现）
- [ ] 修复后新增验证测试，验证修复有效
- [ ] 回归测试：确保不引入新问题
```

**完成标准:** 修复计划文档完成，用户确认

## Phase 3: 实现计划

**调用 skill:** `superpowers:writing-plans`

**沟通程度:** 中

**文档存储:** ⚠️ 参见 `references/document-storage.md`

**步骤:**
1. 计划保存到 `.dev_doc/<bug>/plan.md`
2. 包含每个步骤的：精确文件路径、测试代码框架、运行命令
3. **⚠️ 必须分析并发可行性**：在计划中明确标注哪些步骤可并发执行

**并发可行性分析要求:**
```
## 并发可行性分析

### 可并发步骤（无依赖，可并行执行）
| 步骤 | 依赖关系 | 说明 |
|------|----------|------|
| 步骤A | 独立 | 无文件/数据依赖 |
| 步骤B | 独立 | 无文件/数据依赖 |

### 不可并发步骤（必须串行）
| 步骤 | 依赖关系 | 说明 |
|------|----------|------|
| 步骤C | 依赖步骤A | 需等步骤A完成后才能开始 |
```

**阶段声明:**
```

正在制定详细实现计划...
正在分析并发可行性...
```

**完成标准:** 计划文档完成，包含并发可行性分析

## Phase 3.1 + 3.2: 并行独立 Review（可并发执行）

**沟通程度:** 低（独立 Subagent 执行）

★ Insight ─────────────────────────────────────
Phase 3.1 和 Phase 3.2 可以**并发执行**：
- 两者读取相同的文档（plan.md、fix-plan.md）
- 检查维度不同（完整性 vs 合规性）
- 并行节省时间，且互不影响
─────────────────────────────────────────────────

**触发时机:** Phase 3 计划文档完成后

**并发执行方式:**
```
同时启动两个独立 Subagent（Agent tool）：

Subagent A - 计划完整性审查：
- 角色：刚接手Bug的新开发者
- 只读取：.dev_doc/<bug>/plan.md、fix-plan.md
- 不加载：Phase 1-3 的对话上下文
- 任务：检查计划是否完整、无 TODOs/占位符、步骤清晰可执行

Subagent B - 合规门禁审查：
- 角色：质量审计员
- 只读取：.dev_doc/<bug>/plan.md、fix-plan.md、bug-analysis.md
- 不加载：Phase 1-3 的对话上下文
- 任务：按照合规检查清单逐项审查
```

**Phase 3.1 检查项:**
- [ ] 计划是否完整（无 TODOs、占位符）
- [ ] 步骤分解是否清晰可执行
- [ ] 是否与修复计划对齐
- [ ] 每个修复点都有对应实现步骤
- [ ] 每个修复点都有对应测试计划

**Phase 3.2 检查清单:**

| 检查项 | 通过标准 |
|--------|----------|
| Bug覆盖 | 原Bug的每个问题点都有对应的修复计划 |
| 无过度修复 | 没有修改需求范围之外的功能 |
| 修复路径 | 每个问题点都有明确的修复方案 |
| 风险识别 | 已识别并规划了主要技术风险 |
| 测试计划 | 每个修复点都有对应的测试用例 |
| 影响分析 | 已分析改动对其他模块及上下游的可能影响 |

**阶段声明:**
```
🔍 进入 Phase 3.1 + 3.2: 并行独立 Review
正在同时审查计划完整性和合规性...
```

**Review 失败处理:**
- 任一 Subagent 不通过 → 直接在 plan.md 中调整计划 → 重新执行对应 Review
- ⚠️ **自纠闭环原则**：Review 失败后立即自纠，不要停下来等用户确认
- 两个都通过 → 进入 Phase 4

**完成标准:** 两个独立 Subagent 都给出 ✅通过 结论

## Phase 4: TDD开发

**沟通程度:** 低

**⚠️ Worktree 强制要求**

★ Insight ─────────────────────────────────────
Worktree 隔离是保护主分支不被污染的关键机制。
跳过 worktree = 违反 CRITICAL RULES。
开发完成后 worktree 保留，由用户手动 merge。
并发开发必须为每个 Subagent 创建独立 worktree。
─────────────────────────────────────────────────

**所有开发必须在 worktree 中进行，禁止直接在本地分支开发。**

**决策：根据 Phase 3 的并发可行性分析，选择执行策略：**

### 策略 A：多 Subagent 并行开发（当存在可并发步骤时）
```
1. 为每个并发步骤创建独立 worktree
   git worktree add .worktrees/<bug>-<步骤A> -b fix/<bug>-<步骤A>
   git worktree add .worktrees/<bug>-<步骤B> -b fix/<bug>-<步骤B>

2. 同时启动多个独立 Subagent 并行开发
   Subagent A：
   - 角色：修复开发者
   - 只读取：.dev_doc/<bug>/plan.md（步骤A部分）
   - 工作目录：.worktrees/<bug>-<步骤A>
   - 任务：按照计划执行步骤A的修复

   Subagent B：
   - 角色：修复开发者
   - 只读取：.dev_doc/<bug>/plan.md（步骤B部分）
   - 工作目录：.worktrees/<bug>-<步骤B>
   - 任务：按照计划执行步骤B的修复

3. 各 Subagent 独立提交到各自的 worktree
4. 合并回主 fix worktree
```

### 策略 B：串行开发（当无可并发步骤时）
```
1. 创建单一 worktree
   git worktree add .worktrees/<bug> -b fix/<bug>

2. 按计划顺序执行每个任务
3. 严格遵循 red-green-refactor 循环
4. 频繁提交（每个修复点或2-5步后）
```

**阶段声明（并发模式）:**
```
🔧 进入 Phase 4: TDD开发（并发模式）
正在为 [步骤A]、[步骤B] 创建独立 worktree...
正在启动并行 Subagent 开发...
```

**阶段声明（串行模式）:**
```
🔧 进入 Phase 4: TDD开发（串行模式）
已在 worktree 中开始修复，LLM 将自主执行计划...
```

**状态更新:** 进入 Phase 4 时更新 `.dev_doc/<bug>/status.md`

**任务进度更新（调用 todo_list_manager）:**
```
进入 Phase 4 后，立即调用 todo_list_manager skill：
- 将对应任务状态更新为"进行中"
- 记录当前阶段和预计完成时间
⚠️ 这是一个主动调用，不等待用户确认
```

**完成标准:** 计划中所有修复步骤完成，测试通过

## Phase 4.1: 独立代码综合 Review

**沟通程度:** 低（独立 Subagent 执行）

**触发时机:** Phase 4 开发完成后

**任务:** 启动独立 Subagent 审查 worktree 中的代码质量

**执行方式:**
```
启动独立 Subagent（Agent tool）：
- 角色：代码审计员
- 只读取：.worktrees/<bug>/ 下的代码文件
- 不加载：Phase 1-4 的对话上下文
- 任务：审查代码质量、潜在 bug、测试覆盖、影响分析
```

**检查项:**
- [ ] 代码质量问题（单一职责、错误处理）
- [ ] 潜在 bug 风险
- [ ] 测试覆盖是否充分
- [ ] **影响分析**：改动对其他模块及上下游的影响是否已识别并处理

## Subagent 并发执行场景

★ Insight ─────────────────────────────────────
并发 Subagent 的核心价值：
- 独立执行：subagent 之间无上下文共享，避免互相影响
- 客观视角：没有前序"生成记录"的锚定效应
- 效率提升：并行处理耗时任务
─────────────────────────────────────────────────

| 场景 | 为什么适合并发 | 效益 |
|------|---------------|------|
| Phase 3.1 + 3.2 | 读取相同文档，检查维度不同 | 节省 50% 时间 |
| Phase 4 多独立修复 | 修复步骤之间无依赖 | 并行修复提速 |
| Phase 5 验证审查 | 需要客观视角，避免验证者偏差 | 更可靠的验证结果 |
| Phase 5.1 用例生成 | 需要客观视角，避免生成记录影响 | 更高质量的测试用例 |
| Phase 5.5 用例质量审查 | 审查生成用例质量，独立视角 | 更全面的覆盖率 |

**并发执行原则：**
- 只有相互独立、无数据依赖的 Phase 才能并发
- 并发 Subagent 必须独立启动，不共享上下文
- 任一并发任务失败，需返回重做对应部分

**Phase 3 → Phase 4 决策流程：**
```
Phase 3 计划完成后：
  ├── 存在可并发步骤？ → Phase 4 采用多 Subagent 并行修复
  │   ├── 为每个并发步骤创建独立 worktree
  │   ├── 同时启动多个 Subagent
  │   └── 各 Subagent 独立提交
  │
  └── 无可并发步骤？ → Phase 4 采用串行修复
      ├── 创建单一 worktree
      └── 按计划顺序执行
```
