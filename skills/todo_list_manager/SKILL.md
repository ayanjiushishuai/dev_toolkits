# Todo List Manager Skill

本 skill 负责管理 `.dev_doc/todo.md` 和 `.dev_doc/work_done.md`，提供需求录入、优先级排序、完成验证、工作量估算、变更记录，以及每日、每周、每月计划管理八大核心能力。

## 核心概念

### 任务层级
- **父任务**：模块级别，如 `1.3 数据库 Skill 重构`
- **子任务**：最小开发单元，如 `(1) 封装cli`
- 所有子任务完成后，父任务才算完成

### 优先级体系
| 优先级 | 含义 | 处理顺序 |
|--------|------|---------|
| P0 | 最高优先 | 立即处理 |
| P1 | 高优先 | 尽快处理 |
| P2 | 中优先 | 计划处理 |
| P3 | 低优先 | 有空处理 |

### 时间尺度
| 尺度 | 触发条件 | 范围 |
|------|---------|------|
| 每日 | "今天计划"、"今日任务" | 当天 |
| 每周 | 每周一早上 / "本周计划" | 周一至周五 |
| 每月 | 每月1号 / "本月计划" | 1日至月末 |

### 工作日约束
- **默认只安排工作日（周一至周五）**
- 周六、周日不安排计划
- 如用户明确要求周末工作，按用户要求安排

## 模式一：Todo 更新模式

当用户说"记录需求"、"调整优先级"、"XX完成了"时触发。

### 功能 1: 需求录入 (Add)

当用户提供新需求时：

**Step 1: 解析需求**
从用户描述中提取：
- 任务标题
- 所属模块（匹配 `.dev_doc/` 下目录）
- 优先级建议（用户提供或 AI 建议）
- 预估工作量
- DDL（如果有）

**Step 2: 生成 todo 条目**

格式：
```markdown
- 【P{优先级}] {任务编号}.{子任务编号} {任务标题}
  - 模块：{所属模块}
  - 描述：{任务详细描述}
  - 工作量：{估算时间}
  - DDL：{截止日期，如有}
  - 分支：{当前所在分支名}
  - 添加日期：{YYYY-MM-DD}
```

**Step 3: 更新 todo.md**
- 找到对应模块位置
- 按优先级插入（P0 > P1 > P2 > P3）
- 编号自动递增

### 功能 2: 优先级管理 (Prioritize)

当用户要求查看或调整优先级时，执行综合评分模型。

#### 综合评分模型（满分 100）

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| **业务价值** | 0-30 | 对核心场景的影响程度 |
| **依赖关系** | 0-20 | 是否 block 其他关键任务 |
| **紧迫度** | 0-25 | DDL 或业务时限的远近 |
| **工作量** | 0-15 | 越小越高分（快速胜利原则） |
| **认知清晰度** | 0-10 | 需求理解越清晰越高分 |

#### 评分细则

**业务价值 (0-30)**：
- 核心功能/直接影响收入：25-30
- 重要功能/提升效率：18-24
- 辅助功能：10-17
- 边缘功能：0-9

**依赖关系 (0-20)**：
- 被多个关键任务依赖：18-20
- 被1-2个任务依赖：12-17
- 独立任务：8-11
- 被依赖但可替代：0-7

**紧迫度 (0-25)**：
- DDL 在 3 天内：23-25
- DDL 在 1 周内：18-22
- DDL 在 2 周内：12-17
- 无 DDL 但有时限：6-11
- 无明确时限：0-5

**工作量 (0-15)**：
- 1 天内：14-15
- 2-3 天：11-13
- 1 周：7-10
- 2 周：4-6
- 1 个月以上：0-3

**认知清晰度 (0-10)**：
- 需求明确、可直接开发：9-10
- 需求较明确、有小问题待确认：6-8
- 需求模糊、需要进一步分析：3-5
- 需求不明确、需要重新调研：0-2

#### 输出格式

```markdown
## 优先级分析报告

### 当前任务优先级列表（按综合评分降序）

| 排名 | 任务 | 综合评分 | 业务价值 | 依赖关系 | 紧迫度 | 工作量 | 认知清晰度 | 建议 |
|------|------|---------|---------|---------|--------|-------|-----------|------|
| 1 | xxx | 85 | 28 | 18 | 22 | 12 | 5 | P0 |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

### 优先级调整建议
- 【建议 P0→P1】xxx：原因...
- 【建议新增 P0】xxx：原因...
```

### 功能 3: 完成验证 (Verify)

当 ez-dev 到达 P6 阶段，或用户说"XX完成了"时触发。

#### 验证维度

**1. 代码存在性**
- 检查对应文件是否存在
- 检查分支/commit 是否包含相关实现
- 检查关键函数/类是否已定义

**2. 文档一致性**
- 对比 todo.md 中的任务描述
- 检查 `.dev_doc/{模块}/` 下是否有对应开发文档
- 验证文档与代码的一致性

**3. 功能完整性**
- 检查所有子任务是否都已完成
- 检查是否有遗漏的边界情况

**4. Git Merge 状态（核心判断点）**
- **必须检查任务对应的分支是否已合并到主分支（main/dev_zby）**
- 检查命令：`git log --oneline main..{分支名}` 或 `git branch --merged`
- 如果分支已合并 → 满足条件
- 如果分支未合并但确定无后续操作 → 需要用户确认
- 未合并且有后续计划 → 不算完成

> **核心原则**：是否完成的核心判断点在于是否已合并到主分支。只要分支已合并且无后续操作计划，即视为完成。

#### 验证结论

| 结论 | 含义 | 处理方式 |
|------|------|---------|
| **通过** | 所有验证点都满足 | 移动到 work_done.md |
| **需补充** | 部分验证点未满足 | 列出需要补充的内容 |
| **存疑** | 无法自动判断 | 提请用户人工确认 |

#### 执行流程

```
1. 解析任务：识别需要验证的任务
2. 执行验证：逐项检查验证维度
3. 生成报告：输出验证结果
4. 用户确认：等待用户确认后才能移动到 work_done.md
```

#### 验证报告格式

```markdown
## 任务完成验证报告

### 任务：{任务标题}
### 验证时间：{YYYY-MM-DD HH:mm}

### 验证结果：通过 / 需补充 / 存疑

### 详细验证

#### 1. 代码存在性
- 文件检查：✓/✗ {文件路径}
- 分支检查：✓/✗ {分支名}
- 关键实现：✓/✗ {函数/类名}

#### 2. 文档一致性
- 开发文档：✓/✗ {文档路径}
- 文档更新：✓/✗

#### 3. 功能完整性
- 子任务闭环：✓/✗ {完成情况}

#### 4. Git Merge 状态
- 当前分支：{branch_name}
- 合并状态：✓/✗ 已合并到 {main_branch}
- 判断依据：{git log 或 merge info}

### 下一步操作
{等待用户确认 / 需要补充的内容}
```

### 功能 4: 工作量估算 (Estimate)

#### 估算方法

1. **基于任务类型**：
   - CLI 封装：0.5-1 天
   - Bug 修复：0.5-2 天
   - 工具类开发：1-3 天
   - 模块重构：3-7 天
   - 新功能开发：5-10 天

2. **基于历史参考**：
   - 参考 work_done.md 中类似任务的实际工时
   - 考虑技术栈相似性

3. **基于复杂度**：
   - API 数量
   - 数据迁移需求
   - 测试覆盖要求

#### 输出格式

```markdown
### 工作量估算：{任务标题}

| 估算类型 | 时间 | 依据 |
|---------|------|------|
| 乐观 | {X} 天 | 假设一切顺利 |
| 预期 | {X} 天 | 正常情况 |
| 保守 | {X} 天 | 考虑风险 |

参考历史：
- 类似任务 "{xxx}" 实际用时 {X} 天
```

### 功能 5: 变更记录 (Log)

所有优先级调整自动记录到 work_done.md 的 changelog 区。

#### Changelog 格式

```markdown
## Changelog

### {YYYY-MM-DD}
- 【优先级调整】{任务标题}：{调整内容} → {调整后}
  - 原因：{调整原因}
- 【新增任务】{任务标题} → P{优先级}
- 【任务完成】{任务标题} → 已移动到 work_done.md
```

## 模式二：每日计划模式

当用户说"生成今日计划"、"看看今天该做什么"、"今天计划"时触发。

### 执行步骤

**Step 0: 清理已完成任务（每次必做）**
在生成今日计划之前，先扫描所有待办任务。

#### 检测策略（两阶段）

**阶段 1：有分支信息**
如果任务有 `分支：`字段：
1. 执行 `git branch --merged {main_branch} | grep {分支名}`
2. 如果分支已合并 → 判定为已完成
3. 如果分支未合并 → 进入阶段 2

**阶段 2：无分支信息或阶段 1 未通过**
从任务描述中提取 2-3 个关键词（如"Milvus 批量插入"），执行：
```bash
git log --oneline main --all --grep="{关键词}" -n 5
git log --oneline main --all -S "{关键词}" -n 3
```
- 如果找到候选 commit，执行二次验证：
  ```bash
  git log -n 1 --follow {commit_hash} --name-only
  ```
  检查 commit 是否涉及任务 `模块：`目录下的文件（如 `services/vector_stores/` 对应 vector-store 模块）
  - **改了模块文件** → 判定为已完成
  - **只改了测试文件/文档/无关文件** → 保持待办（不算误判）
  - **改了其他相关文件** → 标记为"存疑"，询问用户
- 如果只在其他分支 → 保持待办

#### 执行流程
1. 对每个未标记"已完成"的任务执行检测
2. **已合并**：追加到 work_done.md，从 todo.md 删除
3. **未合并**：保持原样
4. **无法判断**：标记为"存疑"，询问用户

#### 输出报告
```markdown
### 任务清理报告
- 【已迁移】{任务标题} → work_done.md
- 【未合并】{任务标题} → 仍需开发
- 【存疑】{任务标题} → 无法自动判断，请确认
```

**Step 1: 读取 todo.md**
按优先级排序，列出当前所有 P0 和 P1 任务。

**Step 2: 分析今日可执行任务**
- 排除已 block 的任务
- 考虑任务的依赖关系
- 选择当日最合适的任务组合

**Step 3: 生成今日计划**

```markdown
## 今日开发计划

### 日期：{YYYY-MM-DD} {星期X}

### 今日目标
{一句话描述今日核心目标}

### 任务列表

#### P0 任务（必须完成）
1. 【{任务编号}】{任务标题}
   - 模块：{所属模块}
   - 预计工作量：{X} 天
   - 下一步行动：{具体行动}

#### P1 任务（尽量完成）
2. 【{任务编号}】{任务标题}
   ...

### 参考信息
- 当前开发分支：{git branch}
- 近期完成：{来自 work_done.md 的最近 2-3 项}
- 阻塞事项：{如有}
```

## 模式三：每周计划模式

**自动触发**：每周一早上自动触发（用户首次当天对话时）
**手动触发**：当用户说"本周计划"、"这周计划"、"帮我安排本周工作"时触发。

### 执行步骤

**Step 0: 清理已完成任务（每次必做）**
同每日计划模式，采用两阶段检测策略（分支名优先，其次 git log 关键词搜索），输出任务清理报告。

**Step 1: 确定本周范围**
- 本周一至本周五（工作日）
- 如已过周一，从当天开始计算剩余工作日
- 周六、周日不安排

**Step 2: 分析本周可用任务**
- 从 todo.md 读取所有 P0、P1 任务（优先处理）
- 评估 P2 任务如有空余时间
- 排除已 block 或有外部依赖的任务
- 汇总本周可用工作日数量（每天按 1 天工作量计）

**Step 3: 与用户确认周目标**
询问用户：
- "本周计划完成哪些任务？"
- 解析用户输入，匹配到 todo.md 中的具体任务
- 确认工作量是否与可用时间匹配

**Step 4: 写入 todo.md 并生成每日分解**

更新 todo.md 顶部，插入本周计划区块：

```markdown
## 本周计划

### {YYYY}年第{WW}周 ({起始日期} ~ {结束日期})
**本周目标**：{一句话描述核心目标}
**完成标志**：{任务完成的验证标准}

#### 每日安排

##### {星期X} {月}/{日}
**主攻：{任务名称}**
- 任务：{具体子任务}
- 预计工作量：{X} 小时 / {Y} 天
- 下一步行动：{具体行动}

##### {星期X} {月}/{日}
...

#### 本周任务优先级排序

| 优先级 | 任务 | 计划日 | 所需工时 | 状态 |
|--------|------|--------|---------|------|
| P0 | xxx | 周X | X天 | ⬜ |
| P1 | xxx | 周X | X天 | ⬜ |

#### 风险与调整预案
- {风险点}：{应对方案}

---

```

**Step 5: 每日进度确认**
每天早上（周一至周五）自动询问：
- "昨天任务完成情况如何？"
- 根据用户反馈更新任务状态
- 如有偏差，调整后续计划

## 模式五：独立清理模式

当用户说"清理已完成任务"、"清理 todo"、"同步任务状态"时触发。

### 执行步骤

**Step 1: 扫描所有待办任务**
读取 `.dev_doc/todo.md`，对每个未标记"已完成"的任务：

1. 检查 Git merge 状态：
   - `git branch --merged {main_branch}` — 检查该分支是否已合并
   - `git log --oneline {main_branch}..HEAD` — 当前分支独有的 commits

2. 分类处理：
   - **已合并到 main/dev_zby** → 执行迁移
   - **未合并且有未完成工作** → 保持不动
   - **无法自动判断** → 标记为"存疑"

**Step 2: 执行迁移**
对每个"已合并"任务：
1. 读取任务完整信息（标题、模块、描述、工作量）
2. 追加到 `work_done.md` 对应模块
3. 从 `todo.md` 中删除该任务
4. 重新编号后续任务

**Step 3: 输出报告**
```markdown
## 任务清理报告

### 日期：{YYYY-MM-DD}

| 任务 | 原分支 | 清理结果 |
|------|--------|---------|
| xxx | feature/xxx | ✅ 已迁移到 work_done.md |
| yyy | feature/yyy | ⚠️ 存疑，需人工确认 |
| zzz | - | ⬜ 未合并，保持不动 |

### 操作汇总
- 已迁移：{N} 项
- 存疑：{N} 项
- 保持：{N} 项
```

## 模式六：进度确认与调整

当用户说"进度如何"、"完成了吗"、"调整计划"时触发。
每周计划期间每天早上也会自动触发。

### 执行流程

**Step 0: 清理已完成任务（每次必做）**
同每日计划模式，采用两阶段检测策略（分支名优先，其次 git log 关键词搜索），输出任务清理报告。

**Step 1: 对照本周计划检查任务状态**
- 逐项检查"每日安排"中的任务
- 询问用户完成情况

**Step 2: 更新任务状态**
- 已完成 → 标记为 ✅
- 进行中 → 标记为 🔄
- 未开始 → 保持 ⬜

**Step 3: 如有必要，调整后续计划**

```markdown
## 进度确认报告

### 日期：{YYYY-MM-DD}

### 本周完成情况
- 【已完成 ✅】任务1
- 【进行中 🔄】任务2（80%）
- 【未开始 ⬜】任务3

### 计划偏差分析
- {任务}：原计划X天，实际Y天，偏差Z天
- 原因：{分析}

### 剩余任务
- {任务A}：还需X天，预计周X完成
- {任务B}：还需Y天，预计周X完成

### 调整建议
- 【建议】{调整方案}
```

- 完成后删除对应任务，记录到 work_done.md
- 按业务模块分类，每个模块对应 `.dev_doc/` 下的独立目录，复杂模块可建立多层文档结构
- 所有模块统一按优先级排序，处理优先级：P0 > P1 > P2 > P3
- 每次删除任务后，及时修改任务编号
- 如果没有P0的任务，选一个你觉得最重要的P1任务改为P0

## 触发条件汇总

| 用户说法 | 触发功能 |
|---------|---------|
| "记录这个需求" / "添加一个任务" | Add |
| "看看优先级" / "调整优先级" | Prioritize |
| "XX完成了" / "ez-dev P6" | Verify |
| "估算一下工作量" / "需要多久" | Estimate |
| "生成今日计划" / "今天做什么" / "今天计划" | 每日计划模式 |
| "本周计划" / "这周计划" / 每周一早上 | 每周计划模式 |
| "本月计划" / "这个月计划" / 每月1号 | 每月计划模式 |
| "进度如何" / "完成了吗" / "调整计划" | 进度确认与调整 |
| "清理已完成任务" / "清理 todo" / "同步任务状态" | 独立清理模式 |
| 每周一首次对话 | 自动触发本周进度确认 |

## 注意事项

1. **最小单元是子任务**：不要只更新父任务，要细化到具体可执行的子任务
2. **验证是核心**：完成验证确保代码、文档、实际实现三者一致
3. **用户确认是必须的**：Verify 结论需要用户确认后才能移动到 work_done.md
4. **自动记录变更**：所有优先级调整都要记录到 changelog
5. **中文输出**：所有输出内容使用中文
6. **工作日约束**：默认只安排周一至周五，周末不安排
7. **计划写到 todo.md**：每周/每月计划作为特殊区块写入 todo.md 顶部，过期后移动到 work_done.md
