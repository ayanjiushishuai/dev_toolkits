# 测试规则清单

> 本文件包含可执行的规则和检查清单，可直接用于审核。

---

## 测试点识别规则

### 必须识别的测试点

| 优先级 | 测试点类型 | 识别标准 | 示例 |
|--------|------------|----------|------|
| P0 | 核心业务逻辑 | 直接影响功能正确性 | `calculate_total()` 计算结果 |
| P0 | 数据转换 | 输入→输出的核心变换 | `parse_user_input()` |
| P1 | 边界条件 | 取值范围的边界 | `top_k` 边界值 |
| P1 | 状态转换 | 对象状态变化路径 | `pending → confirmed` |
| P2 | 异常处理 | 错误情况的处理 | 网络超时、数据无效 |
| P3 | 次要路径 |  редко 使用的分支 | 调试日志、冗余校验 |

### 测试点识别检查清单

```
□ 识别核心业务逻辑（必须 P0）
□ 识别输入验证逻辑
□ 识别状态转换路径
□ 识别异常处理分支
□ 识别依赖外部服务的调用
□ 识别并发/竞态场景（如果有）
```

## 断言规则

### 断言层级（从具体到宽松）

| 层级 | 规则 | 示例 |
|------|------|------|
| **L1** | 验证返回值类型 | `assert isinstance(result, dict)` |
| **L2** | 验证核心字段存在 | `assert "id" in result` |
| **L3** | 验证字段类型 | `assert isinstance(result["count"], int)` |
| **L4** | 验证具体值 | `assert result["count"] == 5` |
| **L5** | 验证业务约束 | `assert 0 <= result["score"] <= 1` |

### 断言检查清单

```
□ 返回值类型正确（dict/list/int/str）
□ 核心字段存在（id、status、data 等）
□ 字段类型正确
□ 具体值符合预期
□ 业务约束满足（范围、一致性等）
```

### 禁止的断言

| 禁止 | 原因 | 正确做法 |
|------|------|----------|
| `assert result is not None` | 太宽松 | 验证具体值 |
| `assert result` | 隐式布尔 | `assert result == expected` |
| `assert len(result) > 0` | 不验证内容 | `assert result["data"] == expected` |

## Mock 规则

### Mock 决策

```
需要 Mock：
├─ 外部 HTTP API（不可控）
├─ 时间函数（确定性测试）
├─ 随机数生成（可重复性）
├─ 第三方 SDK（隔离依赖）
└─ 慢速 I/O（加速测试）

不应 Mock：
├─ 项目内部模块（应集成测试）
├─ 数据库（用测试数据库）
├─ 缓存（可启动本地实例）
└─ 核心业务逻辑
```

### Mock 实现检查清单

```
□ Mock 返回合理的默认返回值（不是 None）
□ 异步函数用 AsyncMock
□ Mock 参数可配置
□ async generator 用 lambda 包装
□ Mock 调用验证（assert_called_once_with）
```

## 边界条件规则

### 必测边界

| 边界类型 | 测试值 | 验证点 |
|----------|--------|--------|
| **空值** | `""`, `[]`, `{}`, `None` | 返回值/异常 |
| **最小值** | `0`, `-1`, `MIN` | 下溢/边界 |
| **最大值** | `MAX`, 溢出 | 上溢/截断 |
| **临界值** | `±1` 临界 | 边界行为 |
| **无效类型** | 错误类型传入 | 类型校验 |
| **格式错误** | 错误格式数据 | 校验逻辑 |
| **权限边界** | 无 token/无权限 | 授权检查 |

### 边界测试检查清单

```
□ 空字符串 ""
□ 空数组 []
□ 空对象 {}
□ None
□ 0 和 -1
□ 最大值/溢出
□ 临界值 ±1
□ 无效类型
□ 格式错误
□ 特殊字符
```

## 测试代码规范

### 测试文件结构

```python
import pytest

class Test{ModuleName}:
    """模块测试类"""

    @pytest.fixture
    def setup(self):
        """测试前准备"""
        yield
        """测试后清理"""

    def test_{case_name}(self):
        """
        测试用例: 描述
        优先级: P0/P1/P2
        """
        # Given: 准备测试环境
        # When: 执行被测功能
        # Then: 验证结果
```

### 测试命名规范

| 规则 | 推荐 | 不推荐 |
|------|------|--------|
| 描述性 | `test_user_creation_with_valid_email` | `test_create` |
| 包含条件 | `test_payment_fails_when_balance_insufficient` | `test_fail` |
| 包含结果 | `test_order_status_confirmed` | `test_status` |
| 使用下划线 | 小写 + 下划线 | camelCase |

### 测试代码检查清单

```
□ 测试函数以 test_ 开头
□ 测试命名描述性且包含条件/结果
□ Given-When-Then 模式
□ 每个测试独立运行（不依赖顺序）
□ 测试后清理（fixture/teardown）
□ 使用 fixtures 管理共享数据
□ 添加 @pytest.mark 标记
```

## 失败分类规则

### 失败类型与处理

| 类型 | 特征 | 处理 |
|------|------|------|
| **代码 Bug** | `AssertionError` | 标记用户关注 |
| **测试问题** | `TypeError`, `KeyError` | 自动修复 |
| **配置问题** | `KeyError: 'xxx'` | 修复配置加载 |
| **基础设施** | `ConnectionError`, `Timeout` | 检查服务 |

### 分类决策树

```
测试失败
  │
  ├─ AssertionError？
  │   └─ 是 → 代码 Bug → 标记用户关注
  │
  ├─ TypeError/KeyError？
  │   ├─ Mock 配置错误 → 自动修复
  │   └─ 配置缺失 → 修复配置加载
  │
  ├─ 连接/超时？
  │   └─ 检查服务状态
  │
  └─ 其他？
      └─ 分析原因，分类处理
```

### 失败处理检查清单

```
□ 分析错误类型
□ 区分代码 Bug / 测试问题 / 环境问题 / 基础设施
□ 代码 Bug → 标记需用户关注
□ 测试代码问题 → 自动修复
□ 环境配置问题 → 修复配置加载
□ 基础设施问题 → 检查并报告
□ 修复后执行回归测试
```

```markdown
# 测试报告 - {date}

| 指标 | 数值 |
|------|------|
| 总测试数 | {n} |
| 通过 | {n} |
| 失败 | {n} |

| 通过率 | {n}% |

## 失败分类
| 类型 | 数量 | 处理状态 |
|------|------|----------|

| 测试问题 | {n} | 已修复 |
```

### 报告分级

| 级别 | 受众 | 内容 |
|------|------|------|
| **开发者报告** | 开发者 | Bug 列表 + 修复建议 |

### 报告检查清单

```
□ 包含测试概览（总数/通过/失败/通过率）
□ 失败按类型分类统计
□ 代码 Bug 单独列出（需用户关注）
□ 自动修复的问题列出
□ 开发者报告只包含 Bug（不打扰）
□ 完整报告包含所有数据
```

## 执行分层规则

### 分层定义

| 层级 | 范围 | 执行频率 | 标记 |
|------|------|----------|------|
| Unit | 模块/函数级隔离 | 每次提交 | `@pytest.mark.unit` |
| Integration | 多模块交互 | 每日/每周 | `@pytest.mark.integration` |
| E2E | 完整业务流程 | 里程碑 | `@pytest.mark.e2e` |

### 执行顺序

```
1. Unit 测试（快速反馈）
   └─ pytest -v -m "unit"

2. Integration 测试（验证集成）
   └─ pytest -v -m "integration"

3. E2E 测试（端到端验证）
   └─ pytest -v -m "e2e"
```

### 分层执行检查清单

```
□ 先执行 Unit 测试（快速反馈）
□ Unit 通过后再执行 Integration
□ Integration 通过后再执行 E2E
□ 失败时快速失败（不等待所有测试完成）
□ 标记正确（@pytest.mark.unit/integration/e2e）
```

## 回归测试规则

### 回归范围确定

| 场景 | 回归范围 |
|------|----------|
| 修复后回归 | 失败用例 + 直接相关用例 |
| 增量回归 | 新增/修改代码相关用例 |
| 全量回归 | 所有用例 |

### 回归验证检查清单

```
□ 修复的测试现在通过
□ 直接相关功能测试仍然通过
□ 没有引入新失败
□ 性能没有退化
□ 修复已同步到环境
```
